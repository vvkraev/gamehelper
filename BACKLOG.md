- [ ] **CRAFT-11** Точный учёт потреблённых орбов при overflow-крафте
  - **Проблема**: приложение считает нажатия кнопки («клики»), а не фактические потребления орба. При overflow-механике игра возвращает орб, если пытается заменить суффикс в overflow-слоте — сообщение «невозможно применить». В результате реальное потребление орбов занижено примерно вдвое или более.
  - **Обнаружено**: батч Time-Lost Sapphire 2026-06-26, шаг `chaos_orb_2_good_prefixes` — записано 516 кликов, фактически потреблено ~206–258 орбов.
  - **Решение**: после каждого применения орба считывать буфер обмена (Ctrl+C) и сравнивать хэш текста предмета с предыдущим. Если текст не изменился — орб возвращён, счётчик потребления не увеличивать. Если изменился — орб потреблён, инкрементировать счётчик.
  - **Альтернатива**: слушать системное сообщение об ошибке (звук / визуальный индикатор в игре) — но это менее надёжно, зависит от UI игры.
  - **Влияние на аналитику**: все исторические данные по overflow-сессиям содержат завышенное число орбов; при наличии логов можно пересчитать с коэффициентом ~0.4–0.5.
  - **Зависит от**: ничего; независимое улучшение логирования

- [ ] **CRAFT-9** Верификация механики выбора аффиксов Chaos Orb (открытый вопрос)
  - **Открытый вопрос**: неизвестно, как именно Chaos Orb выбирает новый аффикс. Две конкурирующие модели:
    - **Модель A — единый пул**: орб выбирает из объединённого пула `prefix + suffix` по весам. Тип (prefix/suffix) определяется выпавшим модом. Доля prefix в результате = `ΣweightPrefix / (ΣweightPrefix + ΣweightSuffix)` — не обязательно 50/50.
    - **Модель B — двухшаговая**: сначала выбирается тип (prefix/suffix) с некоторой вероятностью (50/50? пропорционально доступным слотам? пропорционально суммарным весам пула?), затем внутри типа семейство по весу.
  - **Текущая реализация симуляции** (`RunChaosMonteCarloSimulation`) неявно предполагает Модель B с фиксированными `varPrefixes` и `varSuffixes` слотами на каждый бросок. Это одна из возможных правильных моделей, но не верифицированная.
  - **Почему важно для фрактурных предметов**: при фрактурном prefix и Модели A доля новых prefix-аффиксов ≠ 2/5 (слоты) и ≠ 1/2 — она зависит от соотношения суммарных весов пулов. При Модели B с 50/50 вероятность выровнена независимо от слотов.
  - **Как проверить**: в `AffixStatsScanner` добавить отдельный счётчик для фрактурных предметов (`HasFracturedPrefix`, `HasFracturedSuffix`) — отдельный ключ в `AffixStatsData`. После 50+ крафт-сессий на фрактурном предмете измерить реальное соотношение выпавших prefix/suffix модов и сравнить с предсказаниями обеих моделей.
  - **Зависит от**: накопления достаточного числа крафт-сессий на фрактурных предметах

- [ ] **CRAFT-8** Режим крафта с Fractured Orb (фиксация нужного аффикса)
  - **Механика**: Fractured Orb случайно фиксирует один из N обычных аффиксов на предмете (Desecrated и Fractured в выборку не входят). Вероятность попасть в нужный = 1/N.
  - **Типичный сценарий**: предмет 3 обычных + 1 Desecrated → N=3, вероятность 1/3 → ожидание 3 попытки.
  - **Расчёт ожидаемых затрат**:
    ```
    E[cost] = tries_expected × (item_price + preserved_cranium + fractured_orb)
    tries_expected = N (геометрическое распределение, p = 1/N)
    ```
    Пример при N=3, item=6 div, Preserved Cranium=1.85 div, Fractured Orb=X div:
    `E = 3 × (7.85 + X)` → **23.55 + 3X div**
  - **Что нужно реализовать:**
    1. Интерфейс: поле ввода цены предмета-основы, цены Preserved Cranium, цены Fractured Orb, числа аффиксов N → показывать E[cost] в реальном времени
    2. Автоматизация: открыть Well of Souls (Abysal Bone → 3 normal + 1 desecrated), применить Fractured Orb, прочитать буфер → проверить условие остановки (нужный аффикс стал Fractured)
    3. `CraftConditionEvaluator` — добавить проверку `IsFractured = true` для целевого аффикса как условие успеха
    4. При неудаче: сообщить «не тот аффикс заморожен» + счётчик попыток, ожидать следующий предмет-основу
  - **Зависит от**: CRAFT-7 (парсинг Desecrated), CRAFT-4 (модель ожидаемых затрат)

## Торговля и флипинг

- [ ] **TRADE-1** Оценка потенциала флипинга через биржу (OCR)
  - **Контекст**: в PoE2 торговля между игроками происходит через внутриигровой биржевой стакан — заявки на покупку/продажу предметов и орбов, комиссия золотом. Буфер обмена не работает для торгового UI — единственный способ читать данные это OCR с экрана.
  - **Цель**: захватить экран биржи → распознать заявки (предмет, цена, количество) → оценить спред и потенциал флипа → показать рекомендацию
  - **Текущий блокер**: OCR даёт плохое качество на игровом UI — игровые шрифты с контурами/свечением на полупрозрачном фоне, цветные числа, DPI-scaling. Простой Tesseract без предобработки не справляется с цифрами.
  - **Что нужно исследовать:**

- [ ] **TRADE-6** Snapshot-мониторинг листингов профессиональных крафтеров
  - **Идея**: профессиональные крафтеры и стримеры выставляют предметы по рыночной цене, а не произвольно. Если между двумя снэпшотами предмет пропал — он с высокой вероятностью продан. Последняя зафиксированная цена = реальная цена продажи.
  - **Проблема обычного trade API**: торговая площадка показывает листинги, не сделки. Нельзя знать, куплен предмет или снят с продажи.
  - **Предлагаемая архитектура**:
    1. Файл `pro_crafters.json` — список аккаунтов (accountName) профессиональных крафтеров/стримеров
    2. Периодический опрос (раз в N минут) через `GET /api/trade2/search/{league}?account={name}` или аналог
    3. Сохранение снэпшота в `trade_data/snapshots/{date}_{account}.json`
    4. Сравнение снэпшота с предыдущим → пропавший item_id → запись `{item, last_price, likely_sold_at}` в `market_sales_evidence.json`
    5. Агрегация: по классу, iLvl, аффиксам → медианная «реальная» цена (из подтверждённых исчезновений)
  - **Отличие от TRADE-2**: TRADE-2 = собственные продажи (точные данные). TRADE-6 = рыночные данные высокого качества от профи-крафтеров (косвенные, но лучше произвольных листингов).
  - **Уровень доверия**: исчезновение листинга ≠ продажа (могли снять сами). Нужен эвристический фильтр — например, быстрое исчезновение (< 24ч) и без нового листинга того же предмета.
  - **Первый шаг**: вручную собрать список 5-10 аккаунтов профессиональных крафтеров (стримеры PoE2).
  - **Зависит от**: TRADE-3 (инфраструктура торгового API), `trade_data/` директория (уже есть)

- [ ] **TRADE-4** Агрегатный анализ прибыльности по типу базы и редкости
  - **Контекст**: крафт делится на два принципиально разных сценария:
    - **Mass craft синих (полуфабрикаты)**: покупаем N белых баз → крафтим в синие → продаём. Все предметы одного типа с предсказуемым именем (формат: `{prefix} {base} {suffix}`). Средняя прибыль = (Σ продаж − N × стоимость_базы − стоимость_орбов) / M продаж.
    - **Ручной докрафт жёлтых (готовые предметы)**: уникальные имена, разный состав модов — индивидуальный учёт.
  - **Разделение полуфабрикат/готовый**: определяется редкостью. В API: `name = ""` → синий или белый; `name ≠ ""` → редкий. Для синих группировка по `BaseType` даёт однородные партии.
  - **Что нужно реализовать:**
    1. Вкладка или секция «Агрегат» в торговом разделе: сгруппировать записи по `(BaseType, Rarity)`
    2. Для каждой группы: количество продаж · средняя цена · суммарный крафт-cost · суммарный база-cost · суммарная прибыль · средняя прибыль на предмет
    3. Фильтр по периоду (последние 7/30 дней) чтобы сравнивать разные партии крафта
    4. Опционально: ввод средней стоимости базы на уровне группы (применяется ко всем записям без явного `BaseCostDiv`)
  - **Зависит от**: TRADE-3 (данные уже есть), `BaseCostDiv` в `SaleRecord` (добавлен)
