## Крафт и статистика аффиксов

- [x] **CRAFT-1** Добавить выбор подтипа предмета в UI крафта
  - После выбора класса (Body Armours, Gloves, Helmet, Boots) показывать dropdown с подтипом базы
  - Подтипы: Armour (Str), Evasion (Dex), Energy Shield (Int), Armour/Evasion, Armour/Energy Shield, Evasion/Energy Shield
  - Подтип влияет на пул аффиксов библиотеки — исключает аффиксы чужих подтипов
  - Нужно для корректного расчёта вероятности и чистоты статистики весов
  - Связано с: `AffixSubClass`, `AffixStatsData`, `feat/ilvl-orb-tier-weights`

- [x] **CRAFT-2** Добавить `ItemSubType` в сбор статистики крафта
  - Вычислять подтип из `ParsedItem.Requirements` (Str/Dex/Int → Armour/Evasion/Energy Shield)
  - Изменить ключ `AffixStatsData.PerClass` на `"ItemClass|SubType"` (версия 4)
  - Без разбивки по подтипу статистика Body Armours смешивает несовместимые пулы аффиксов

- [x] **CRAFT-3** ~~Перейти с частотной статистики на весовые вероятности~~ — superseded CRAFT-4+CRAFT-6
  - Цель достигнута иначе: «Теория» (CRAFT-4) использует `BuildWeightPool` → poe2db веса → Monte Carlo;
    «Практика» (CRAFT-6) хранит наблюдаемые частоты как параллельный источник — оба показываются одновременно.
  - Нерешённый остаток: формула «~N попыток» для Aug+Annul семантически отличается от Chaos — отдельная задача при необходимости.

- [x] **CRAFT-4** Расчёт вероятности и стоимости для Chaos Orb
  - **Механика**: Chaos Orb выбирает один случайный аффикс из N на предмете (равновероятно, 1/N), затем заменяет его на новый случайный из пула с учётом весов
  - **Реализация**: Monte Carlo симуляция 5000 прогонов — `RunChaosMonteCarloSimulation` в `CraftConditionWindow.xaml.cs`
    - Флаг `OrbCraftProperties.ReplacesOneExisting = true` для Chaos/Greater Chaos/Perfect Chaos
    - Rejection sampling + binary search по кумулятивным весам — O(log n) на выбор
    - Старт с рандомного предмета 3P+3S из пула; условие проверяется напрямую по `AffixName`
    - Асинхронный расчёт: `StartChaosMonteCarloAsync` показывает «расчёт...» пока не готово
    - Отображение: «Теория (poe2db, Monte Carlo): ~N Chaos Orb · ~X.XX div»

- [x] **CRAFT-5** Поддержка рун (патч Legacy of Aldur)
  - **Контекст**: руны при вставке в предмет добавляют новые аффиксы в пул выпадения; poe2db показывает для них вес=1, что не соответствует реальности
  - **Известный пример**: `[Spears] "Thrud's" T1` — Prefix Modifier, стат `"(25–30)% increased Explicit Speed Modifier magnitudes"` (poe2db добавляет суффикс `— Unscalable Value`, т.е. ролл не зависит от ilvl). Единственная запись в `affix_library.json` без `familyId` — скраппер не назначил его, т.к. рунные аффиксы не входят в стандартный тировый пул
  - Хранить данные рун в отдельном `rune_affix_overrides.json` — не изменять `affix_library.json`
  - Для каждой руны: поддерживаемые классы предметов + добавляемые аффиксы + их реальные веса
  - В UI: поле «вставлена руна» при расчёте вероятности — аффиксы руны подмешиваются в пул
  - `ItemParser` встретит `— Unscalable Value` в тексте стата — убедиться что парсер игнорирует этот суффикс при матчинге шаблона
  - Тестовое покрытие рунных аффиксов — отдельный кейс, т.к. они не попадают в `WeakestTierPerFamilyAndClass` (нет `familyId`)
  - Проблема весов poe2db (везде 1) решается через CRAFT-6 — экспериментальные веса перекрывают скрапнутые
  - Зависит от: CRAFT-6

  - **Формат в буфере**: `{ Desecrated Prefix Modifier "Antelope's" (Tier: 1) — Evasion }` — отличается ключевым словом `Desecrated` перед типом модификатора

    5. `affix_library.json` / отдельный файл — база Desecrated-аффиксов (имена, тиры, веса, допустимые классы)

## Торговля и флипинг

- [ ] **TRADE-1** Оценка потенциала флипинга через биржу (OCR)
  - **Контекст**: в PoE2 торговля между игроками происходит через внутриигровой биржевой стакан — заявки на покупку/продажу предметов и орбов, комиссия золотом. Буфер обмена не работает для торгового UI — единственный способ читать данные это OCR с экрана.
  - **Цель**: захватить экран биржи → распознать заявки (предмет, цена, количество) → оценить спред и потенциал флипа → показать рекомендацию
  - **Текущий блокер**: OCR даёт плохое качество на игровом UI — игровые шрифты с контурами/свечением на полупрозрачном фоне, цветные числа, DPI-scaling. Простой Tesseract без предобработки не справляется с цифрами.
  - **Что нужно исследовать:**
    1. Предобработка скриншота: crop нужной области → grayscale → адаптивный threshold → масштабирование → Tesseract с `--psm 7` / `--psm 6` для чисел
    2. Альтернатива Tesseract: Windows.Media.Ocr (встроенный в Windows 10+, работает офлайн, лучше справляется с нестандартными шрифтами)
    3. Шаблонное распознавание: найти UI-элементы через template matching (OpenCV) → вырезать только поля с числами → OCR узкой области
    4. Захват окна: `PrintWindow` / `BitBlt` для точного снимка без артефактов масштабирования
  - **Зависит от**: понимания структуры торгового UI (какие области читать, формат заявок)

- [ ] **TRADE-2** История продаж с анализом ценности модификаторов
  - **Цель**: сохранять каждую продажу (предмет + все его моды + цена) для последующего анализа — какие модификаторы коррелируют с высокой ценой
  - **Исследование `Client.txt`** (Steam: `steamapps/common/Path of Exile 2/logs/Client.txt`, 1.8M строк):
    - `Ange: Sold it!` / `Ange: Good news!` — подтверждение продажи (несколько вариантов фразы). Только **временна́я метка**, без деталей предмета и цены
    - `Ange: I'll do what I can to sell it for you.` — выставление предмета на продажу
    - `Ange: Price drop!` — автоматическое снижение цены
    - **Вывод**: лог даёт надёжный сигнал о факте и времени продажи, но не знает что/за сколько
  - **Предлагаемая архитектура** (два шага):
    1. **При выставлении**: пользователь нажимает Ctrl+C на предмете → приложение перехватывает буфер и сохраняет `{timestamp, ParsedItem, price}` в очередь листингов; цену вводит вручную или через OCR (TRADE-1)
    2. **При продаже**: мониторинг `Client.txt` в реальном времени (tail) → обнаружение `"Ange: Sold it!"` → помечаем ближайший незакрытый листинг как проданный
  - **Хранение**: локальная база (JSON или SQLite) — запись: дата листинга, дата продажи, ItemClass/iLvl/аффиксы, цена
  - **Аналитика**: группировка по аффиксам → средняя цена предметов с данным модом → ранжирование «какой мод даёт наибольшую премию»
  - **Зависит от**: TRADE-1 (OCR для цены), либо ручной ввод цены как первый шаг
