# Механика выбора тира аффикса орбами

Документ описывает как орбы выбирают аффиксы и тиры при крафте, включая
ограничения по ilvl, веса тиров, свойства орбов и ограничения по типу базы.

---

## Алгоритм выбора (3 шага)

### Шаг 1 — выбор слота
Равновероятно **50/50**: Prefix или Suffix.

### Шаг 2 — выбор семейства
Из пула всех семейств, допустимых для данного слота **и данного подтипа предмета**,
выбор производится пропорционально **сумме весов всех тиров** семейства.

```
P(семейство) = Σвесов_тиров_семейства / Σвесов_всех_семейств_слота
```

Следствие: семейство с 8 тирами по 100 весов каждый (Σ=800) выбирается в 8 раз
чаще, чем семейство с 1 тиром весом 100 (Σ=100). Именно поэтому "to Spirit"
на Body Armour встречается существенно реже, чем "increased Energy Shield".

### Шаг 3 — выбор тира
Из допустимых тиров выбранного семейства — выбор по **индивидуальным весам**.
T1 обычно имеет меньший вес, чем T2..Tn, но есть семейства где все тиры
равновероятны. Точные веса неизвестны — нужно собирать статистику.

Фильтр допустимых тиров:
```
tier.modifier_level <= item.ilvl
AND tier.modifier_level >= orb.min_modifier_level
```

**Исключение T1**: если ни один тир семейства не проходит фильтр
(все `modifier_level < orb.min_modifier_level` или все > `item.ilvl`),
автоматически выбирается **T1 данного семейства** независимо от его modifier_level.

### Итоговая вероятность конкретного тира

```
P(тир) = 0.5
       × (Σвесов семейства / Σвесов всех семейств слота для данного подтипа)
       × (вес тира / Σвесов допустимых тиров семейства)
```

## Ограничения по типу базы (ItemSubType)

Не все семейства доступны на всех предметах одного класса. Внутри класса
"Body Armours" существует 6 подтипов по требованиям к характеристикам,
каждый с собственным пулом семейств:

| Подтип | Атрибут | Пример семейства (prefix) |
|---|---|---|
| Armour | Str | % increased Armour |
| Evasion | Dex | % increased Evasion Rating |
| Energy Shield | Int | % increased Energy Shield |
| Armour/Evasion | Str+Dex | % increased Armour and Evasion Rating |
| Armour/Energy Shield | Str+Int | % increased Armour and Energy Shield |
| Evasion/Energy Shield | Dex+Int | % increased Evasion and Energy Shield |

Аналогичное разделение может существовать у других классов предметов
(например, Shields). Пул семейств на шаге 2 всегда берётся для конкретного
`ItemClass` + `ItemSubType`.

## Свойства орбов

| Орб | min_modifier_level | Примечание |
|---|---|---|
| Chaos Orb | 1 | Реролл всех модов, все тиры с level ≥ 1 доступны |
| Perfect Chaos Orb | 50 | Реролл всех модов, только тиры ≥ 50 (или T1 по исключению) |
| Exalted Orb | 1 | Добавляет один мод |
| Aug Orb | 1 | Добавляет мод (на магическом предмете) |
| Regal Orb | 1 | Добавляет один мод (магический → редкий) |
| Divine Orb | — | Не выбирает тир, перебрасывает значение в рамках текущего тира |
| Annulment Orb | — | Удаляет случайный мод, тир не выбирает |

*Свойства дополнять по мере изучения механики других орбов.*

## Данные: что есть и что нужно

| Данные | Источник | Статус |
|---|---|---|
| `modifier_level` (ilvl требование тира) | poe2db scraper → `affix_library.json` | **Есть** (`AffixTierLevel`) |
| Группировка тиров в семейства | Неявно через `AffixStats` + `ItemClass` + `ItemSubType` | **Есть частично** (без SubType) |
| `weight` тира | Нет открытых источников | **Нужно собирать** из статистики крафта |
| `min_modifier_level` орба | Описание предмета в игре | **Нужно задать вручную** |
| `ItemSubType` аффикса | Не в скрэпнутых данных | **Нужно добавить вручную** или через доп. скрэпинг |

## Изменения в модели данных

### `AffixLibraryEntry` — новые поля

```csharp
/// <summary>
/// Вес тира при выборе орбом (шаг 3).
/// null = неизвестен, заполняется из статистики крафта.
/// </summary>
public int? Weight { get; set; }

/// <summary>
/// Подтип базы предмета, ограничивающий пул семейств (шаг 2).
/// Например: "Armour", "Evasion", "Energy Shield", "Armour/Evasion" и т.д.
/// null = аффикс доступен на всех подтипах данного ItemClass.
/// Для Tablet уже используется (Ritual, Breach, и т.д.).
/// </summary>
// Используется существующее поле AffixSubClass
```

Поле `AffixSubClass` уже существует в `AffixLibraryEntry` — его семантику
расширяем: для Body Armours значения "Armour" / "Evasion" / "Energy Shield" /
"Armour/Evasion" / "Armour/Energy Shield" / "Evasion/Energy Shield".

### Новый класс `OrbCraftProperties`

```csharp
/// <summary>
/// Свойства орба, влияющие на выбор тира аффикса.
/// </summary>
public sealed class OrbCraftProperties
{
    public string Name { get; set; } = "";

    /// <summary>
    /// Минимальный modifier_level тира для включения в пул выбора (шаг 3).
    /// 0 или 1 = нет ограничения снизу.
    /// </summary>
    public int MinModifierLevel { get; set; } = 1;

    /// <summary>
    /// Выбирает новый тир аффикса (false для Divine, Annul).
    /// </summary>
    public bool SelectsTier { get; set; } = true;
}
```

### Семейство аффикса (логическая группа)

Явного класса `AffixFamily` не вводим. Семейство определяется как группа
записей `AffixLibraryEntry` с одинаковыми `AffixStats` (шаблоны) +
`ItemClass` + `AffixSubClass` + `AffixType` (Prefix/Suffix).
Вычисляется на лету при расчёте вероятностей.

Сумма весов семейства:
```csharp
family.Entries.Where(e => IsEligible(e, item, orb)).Sum(e => e.Weight ?? defaultWeight)
```

## Сбор статистики весов

Единственный способ узнать веса — собирать данные из реальных крафт-сессий.
Для каждой попытки крафта записываем:
- `ItemClass` + `ItemSubType`
- `item.ilvl`
- Какое семейство выпало (из `AffixStats`)
- Какой тир выпал (`AffixTier`)

По накопленной выборке можно оценить:
- Относительные веса семейств (через частоты на шаге 2)
- Относительные веса тиров внутри семейства (через частоты на шаге 3)

Минимальная выборка для статистически значимых выводов: ~200 попыток
на одну комбинацию `ItemClass` + `ItemSubType`.
