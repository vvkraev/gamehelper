# Учёт стоимости крафта в золоте: поиск текста на экране, клик и журнал

## 1. Цель

Нужна подсистема на основе **компьютерного зрения / OCR**, которая:

1. **Ищет на экране заданный текст** (или близкий к нему — с учётом шрифта, масштаба UI, шума).
2. **Выполняет клик** в найденной области (обычно по центру бокса текста или по настраиваемому смещению).
3. **Записывает событие в файл** в стабильном машиночитаемом формате.

Практическое применение в GameHelper: **оценивать стоимость разных крафт-проектов в золоте** по тому, что отображается в игре или в сопутствующем UI. Отдельные **сценарии** (рецепты, способы крафта, вкладки торговли и т.д.) запускаются независимо; по накопленному журналу позже строится **сводная таблица** — как по дням менялась золотая оценка выбранных способов крафта.

Документ фиксирует **что** строим и **зачем**; детали реализации (библиотека OCR, точность, UI) уточняются по мере внедрения.

---

## 2. Пользовательский сценарий (сквозной)

1. Пользователь один раз настраивает **сценарий** (имя, область экрана или полный экран, шаблон текста или список меток, правила клика, опционально задержки).
2. Запускает сценарий вручную или из существующего потока автоматизации (по аналогии с другими крафт-сервисами).
3. Приложение делает снимок нужной области, **распознаёт текст и его координаты**, сопоставляет с целевой строкой (точное совпадение, регистронезависимость, допуск опечаток — уровни настраиваются).
4. При успешном совпадении: **клик** (ЛКМ/ПКМ по настройке), затем **запись строки в журнал**.
5. При неуспехе: политика повтора (N попыток, интервал), запись в журнал отказа или только в отладочный лог — по настройке.

Итог: файл с историей наблюдений, пригодный для Excel / Power BI / скриптов без привязки к конкретной игре.

---

## 3. Что именно пишем в файл

Журнал должен позволять:

- фильтровать по **дате** и **сценарию**;
- строить **динамику по дням**;
- сравнивать **несколько способов крафта** (разные `scenario_id` или `craft_method`).

**Принято:** журнал в формате **CSV**, **один общий файл** на все сценарии (различие сценариев — колонки `scenario_id` / `scenario_name`; в Excel / сводных таблицах фильтруете по ним). Имя файла по умолчанию, например `craft_gold_cost_log.csv`, папка — как у остальных логов (`Log/` рядом с приложением), путь при необходимости настраивается.

**Первая строка файла — заголовок** (ровно один раз при создании файла), далее **каждая строка = одно событие**. Кодировка: **UTF-8** (с BOM по желанию для удобного открытия в Excel на Windows).

**Порядок и смысл колонок** (заголовок в том же порядке):

| Колонка | Назначение |
|---------|------------|
| `timestamp_utc` | Время события (ISO 8601, например `2026-04-13T14:30:00Z`). |
| `scenario_id` | Стабильный идентификатор сценария (без запятых и кавычек в значении — проще для CSV). |
| `scenario_name` | Человекочитаемое имя. |
| `status` | `ok` / `not_found` / `ambiguous` / `error` и т.д. |
| `matched_text` | Что реально распознали или выбранный фрагмент. |
| `gold_value` | Число в золоте, если извлекается; иначе пусто. |
| `raw_ocr_snippet` | Короткий контекст для отладки; может быть пустым. |
| `screen_rect` | Область поиска в одной ячейке, например `x,y,w,h` без пробелов; может быть пусто. |
| `click_point` | Координаты клика, например `x,y`; может быть пусто. |
| `notes` | Комментарий; может быть пусто. |

**Экранирование:** если в значении есть запятая, перевод строки или кавычка — поле оборачивается в двойные кавычки, внутренние `"` удваиваются (стандарт CSV, как в Excel при экспорте). Пустое поле — просто пусто между запятыми.

**Версия схемы:** номер версии набора колонок хранится в этом документе (сейчас **schema 1**); при добавлении колонок в будущем — увеличить версию в документе и при необходимости добавить новый файл или миграцию — по отдельному решению.

## 4. Архитектура (логические модули)

Чтобы не смешивать OCR, ввод и UI, разумно разделить:

1. **Захват экрана** — снимок `Bitmap` по `ScreenRect` (переиспользование идей из `RegionPickerWindow` / существующих регионов).
2. **Детекция текста** — OCR-движок + возврат **bounding boxes** для строк/слов (обязательно для клика по найденному тексту).
3. **Сопоставление** — сравнение распознанных строк с целевым шаблоном (подстрока, regex, порог похожести).
4. **Ввод** — программный клик в координатах экрана (согласованность с DPI и несколькими мониторами).
5. **Журнал** — потокобезопасная дозапись в **один общий** CSV-файл, ротация по дате (опционально, если файл разрастётся).
6. **Сценарий** — именованная конфигурация: регион, шаблоны, политика повторов, имя метода крафта для отчётов.

Связь с текущим GameHelper: настройки можно хранить в `AppSettings` / `settings.json` по тому же принципу, что области орба и предмета; отдельная вкладка или окно — на этапе UI.

---

## 5. Требования и ограничения

- **Windows + WPF** — приоритет нативным API и зависимостям, совместимым с `net10.0-windows`.
- **Производительность:** OCR тяжелее простого сравнения буфера обмена; для «раз в минуту» сценариев достаточно; для плотного цикла — кэширование и ограничение частоты.
- **Надёжность:** масштаб Windows, HDR, сглаживание шрифтов влияют на качество; в документе закладываем **настраиваемый регион** и **повторы**, а не «идеальный OCR всегда».
- **Безопасность:** клики по экрану — только в явно заданных сценариях; пользователь должен понимать область действия.

## 6. Не в scope первой версии (по умолчанию)

- Построение сводной таблицы **внутри** приложения (достаточно экспорта CSV для внешних инструментов).
- Обучаемые нейросети под конкретную игру (если не потребуется позже).
- Полная интеграция с каждым существующим крафт-сервисом без отдельного явного вызова сценария.

## 7. Поэтапный план развития (черновик)

Этапы можно уточнять после выбора OCR-движка и первого пилота на реальном UI.

| Этап | Содержание | Результат |
|------|------------|-----------|
| **0** | Формат журнала: **CSV**, общий файл, заголовок, колонки (см. §3) | Спецификация зафиксирована; при реализации — пример строки в коде/тесте |
| **1** | Захват региона + вызов OCR + вывод распознанного текста в отладочное окно/лог без клика | Доказательство, что текст читается |
| **2** | Поиск целевой строки по bbox → один клик → одна запись в журнал | Минимально жизнеспособный сценарий |
| **3** | Именованные сценарии в настройках, ручной запуск из UI | Повторяемое использование |
| **4** | Извлечение числа золота из распознанной области (regex / второй ROI) | Поле `gold_value` в журнале |
| **5** | Повторы, таймауты, статусы `not_found` / `ambiguous`, опциональная ротация логов | Стабильность в «боевом» режиме |
| **6** | (Опционально) Триггер из существующих сценариев крафта или планировщик по времени | Меньше ручных действий |

Дальнейшие шаги: **этап 0 закрыт** (CSV + общий лог). Переходить к **этапу 1** — захват региона и OCR без клика.

## 8. Принятые решения и справка

### 8.1. Уже согласовано

- Формат строки: **CSV** (см. §3).
- Журнал: **один общий файл** для всех сценариев.

### 8.2. Что такое NuGet и Tesseract (простыми словами)

- **NuGet** — это стандартный для .NET способ **подключать готовые библиотеки** к проекту: в файле проекта указывается пакет, и при сборке подтягиваются нужные файлы. Это не отдельная программа, которую нужно ставить вручную; обычно всё делает среда разработки (Visual Studio / Cursor) при сборке. Аналог: «менеджер пакетов» как для приложений на телефоне, только для кода.

- **Tesseract** — это **движок распознавания текста с картинки** (OCR), изначально отдельная программа/библиотека. В .NET его часто подключают **через NuGet** плюс файлы обученных языков. Даёт хороший результат в разных условиях, но добавляет зависимости и настройку языков.

**Ориентир для GameHelper:** сначала имеет смысл попробовать **встроенное распознавание Windows** (API вроде Windows Runtime OCR — без NuGet и без Tesseract). Если на вашем UI качество будет слабым, можно отдельно решить подключать Tesseract через NuGet. От вас для старта ничего знать про NuGet не требуется — это уровень разработки при сборке проекта.

### 8.3. Остаётся уточнить позже

1. Источник цены: только **OCR с экрана** или иногда данные из **буфера обмена** / файла?
2. Язык текста в игре (латиница / кириллица / смесь) — влияет на настройку распознавания.
