## 1. Введение

### 1.1. Назначение документа

Зафиксировать требования к приложению GameHelper, чтобы любой участник (или вы через время) мог понять границы продукта без пересказа из чатов.

### 1.2. Область применения

Десктопное приложение под Windows на WPF (.NET), см. целевой фреймворк в `GameHelper.csproj`. Взаимодействие с игрой — через заданные пользователем области экрана, клавиатуру и буфер обмена.

### 1.3. Термины

| Термин | Определение |
|--------|-------------|
| Область экрана | Прямоугольная зона (или пара координат), задаваемая пользователем для кликов или фокуса курсора. |
| Chaos Orb / сфера хаоса | Валюта в PoE2; в сценарии — объект в инвентаре/сташе, по которому выполняется Shift+ПКМ для применения к предмету. |
| Аффикс | Модификатор на предмете; в требованиях задаётся **шаблоном** — либо фрагмент подстроки, либо текст с одной строчной **`n`** на месте группы цифр в буфере. |
| Попытка / операция | Один полный цикл: Shift+ПКМ по области орба → ЛКМ по области предмета → проверка текста из буфера после Ctrl+C. |
| Искомый текст / шаблон | Строка поиска: без **`n`** — проверка вхождения подстроки в буфер **без учёта регистра**; с **`n`** — в буфере ищется совпадение «текст до `n` + одна или более цифр + текст после `n`», затем число сравнивается с **минимальным порогом**, заданным пользователем отдельно. |

## 2. Общее описание

### 2.1. Цель продукта

**GameHelper** автоматизирует процесс **создания (перекрафта) предметов** в **Path of Exile 2**: многократное применение **Chaos Orb** к одному предмету с автоматической проверкой появления **нужного аффикса** в тексте предмета, без ручного повторения одних и тех же кликов до сотен раз.

Пользователь — игрок PoE2, который целенаправленно гонит редкий аффикс хаосами и готов заранее задать области интерфейса и критерий остановки.

### 2.2. Пользователи и сценарии

| Пользователь | Основные сценарии |
|--------------|-------------------|
| Игрок PoE2 | **Основной сценарий «хаос-крафт под аффикс»:** пользователь задаёт **область экрана** через **полноэкранный выделитель** (кнопки «Задать область» для орба и предмета): перетаскивание ЛКМ по всем мониторам; координаты показываются **только для просмотра**, без ручного редактирования. Затем задаёт **шаблон аффикса** (см. термины) и при необходимости **минимальный порог числа** для режима с **`n`**, а также **максимальное число повторов** операции. Далее приложение в цикле выполняет: **зажать Shift** → **ПКМ** в области орба → **ЛКМ** в области предмета → с курсором в области предмета нажать **Ctrl+C**. Затем берёт текст из **буфера обмена** и проверяет условие по FR-005. Если условие **выполнено** — **остановить** цикл. Иначе, если число операций **меньше** максимума — **повторить**. Если достигнут **максимум операций** — **остановить** цикл. **Настройки** (области, шаблон, пороги, задержки, флаги) **сохраняются** в файл `settings.json` в корне проекта (рядом с `.csproj`) при закрытии окна и при изменении областей / после успешного запуска крафта. |

### 2.3. Зависимости и среда

- ОС: Windows (WPF).
- Среда выполнения: согласно `TargetFramework` в проекте.
- Игра PoE2 запущена; окно/интерфейс расположены так, что заданные области остаются валидными на время цикла.
- Пользователь самостоятельно соблюдает правила игры и ToS; приложение лишь автоматизирует локальный ввод.

## 3. Функциональные требования

| ID | Требование | Приоритет | Статус |
|----|------------|-----------|--------|
| FR-001 | Пользователь задаёт **область для Chaos Orb** через кнопку и **выделение мышью** на экране; в UI отображаются X, Y, W, H и точка клика **только для информации** (редактирование координат вручную не требуется и не предусмотрено). | Must | |
| FR-002 | Аналогично FR-001 для **области целевого предмета** (ЛКМ применения орба; фокус для Ctrl+C). | Must | |
| FR-003 | Пользователь может ввести **шаблон аффикса**: произвольный текст; ровно одна строчная **`n`** обозначает место для **одной группы цифр** `(\d+)` в буфере; без **`n`** — режим **подстроки** (см. FR-005). Отдельное поле — **минимальное целое** для сравнения с найденным числом (используется только если в шаблоне есть **`n`**). | Must | |
| FR-004 | Пользователь может задать **целое число N — максимум операций** в одном запуске цикла. | Must | |
| FR-005 | После Ctrl+C: взять текст буфера. **Режим без n:** «аффикс найден», если буфер **содержит** шаблон как подстроку (**без учёта регистра**). **Режим с n:** построить по шаблону регулярное выражение «экранированный текст до n + `(\d+)` + экранированный текст после n», искать в буфере (**IgnoreCase**); при совпадении взять число из группы; «аффикс найден», если **число ≥** заданного минимума. Иначе — продолжать цикл. | Must | |
| FR-006 | Одна **операция** последовательно: удерживать **Shift** → **ПКМ** по центру/внутри области орба → **ЛКМ** по области предмета → **Ctrl+C** при курсоре в области предмета (координаты кликов — по правилам UI, см. проектирование). | Must | |
| FR-007 | При «аффикс найден» — немедленно **остановить** цикл и отчитаться пользователю (статус в UI / сообщение). | Must | |
| FR-008 | При достижении **N операций** без успеха — **остановить** цикл и отчитаться пользователю. | Must | |
| FR-009 | Пользователь может **запустить** и **остановить** цикл с экрана приложения (аварийная остановка). | Must | |
| FR-010 | После **каждого завершённого запуска** крафта сохраняется отдельный файл в каталоге **`Log`** в **корне проекта** (рядом с `.csproj`). В файле по каждой попытке: полный текст **буфера обмена**, **шаблон**, **min** (если применимо), **пояснение проверки**, **успех/продолжение**. Имя: `craft_год-месяц-день_час-минута-секунда--год-месяц-день_час-минута-секунда.txt` (двоеточия в имени заменены дефисами). Сессионный лог UI в том же каталоге (`session_…`) дополняется при закрытии приложения. | Must | |

## 5. Модель данных / интеграции

- **Ввод пользователя:** прямоугольники двух областей (орб, предмет) из UI выделения, шаблон аффикса, при необходимости минимальный порог для `n`, целое N, задержки мыши и буфера, флаги отладки.
- **Внешние системы:** ОС Windows (мышь, клавиатура, `Clipboard`).
- **Логи:** каталог **`Log`** в корне проекта (поиск папки с `.csproj` от `AppContext.BaseDirectory`). Файл крафта: `craft_*_wip.tmp` → `craft_{старт}--{конец}.txt` сразу после завершения цикла. Сессия: `session_*_wip.tmp` → `session_{старт}--{конец}.txt` при выходе из приложения. Формат меток времени в имени файла: `yyyy-MM-dd_HH-mm-ss`.
- **Файл конфигурации:** `settings.json` в корне проекта (обнаружение корня — по наличию `.csproj` вверх от `AppContext.BaseDirectory`), формат JSON (camelCase).

## 6. Критерии приёмки (MVP цикла)

- Заданы области, строка и N; после старта выполняется последовательность FR-006 хотя бы один раз без падения приложения.*
- При подставленном тестовом буфере (или мок) проверка FR-005 даёт ожидаемое «найден / не найден».
- Цикл завершается по **успеху** или по **исчерпанию N**, и пользователь видит итоговый статус.
- «Стоп» прерывает ожидание и не оставляет залипший Shift/кнопки (по возможности ОС — отпускание модификаторов при остановке).

\*Реальный e2e с игрой — в среде пользователя.
